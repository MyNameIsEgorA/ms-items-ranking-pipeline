Предобработка данных и ранжирование товаров по бизнес правилам
Контекст

Вы получили несколько выгрузок из систем заказчика. Данные частично загрязнены, имеют разный уровень детализации и требуют предобработки перед анализом и последующим использованием в сервисе персонализации и поиска.

Вам нужно подготовить данные, связать сущности между собой, продумать временную логику анализа и построить прототип ранжирования товаров для категорийных страниц магазина.

    очищенные и согласованные данные

Требования к подготовке данных
1. Предобработка

Нужно:

    найти и описать проблемы в данных

    очистить и нормализовать данные

    сопоставить сущности между выгрузками

    обработать пропуски, дубли, неоднозначные связи и возможные конфликты идентификаторов

    задокументировать принятые допущения

2. Схема сущностей

Нужно построить и описать связи между сущностями, например:

    товар

    категория

    клиент

    событие

    временной срез или дата наблюдения

Схема должна учитывать временной аспект: на момент расчета нельзя использовать события из будущего.
3. Временное разделение

Нужно предусмотреть механизм временного разделения событий для анализа персонализации в ретроспективе.

Пример:

    все признаки для ранжирования на дату T должны быть рассчитаны только по данным, доступным до даты T

    итоговый результат нужно показать на следующий день после последнего события в датасете

Прототип скоринговой системы на основе правил

Ниже приведен пример возможного подхода. Вы можете использовать его как основу, доработать или частично изменить, если обоснуете свои решения.
Примеры коэффициентов бустинга

BOOST_IN_STOCK         = 1.5      # товар в наличии
BOOST_FEATURED         = 1.3      # ручное поднятие товара
BOOST_SALE             = 1.2      # товар участвует в акции
BOOST_OUT_OF_STOCK     = 0.05     # штраф за отсутствие в наличии
BOOST_CATEGORY_*       = 1.0–1.4  # ручной коэффициент для категории

POPULARITY_HALF_LIFE   = 30       # дней, уточняется после аудита
POPULARITY_WINDOW      = 30       # дней, уточняется после аудита
POPULARITY_W_VIEWS     = 0.3

POPULARITY_W_PURCHASES = 0.7

NOVELTY_WEIGHT         = 1.0

Пример расчета скоринга

popularity  = (views × W_VIEWS + purchases × W_PURCHASES) × e^(−λ × age_days)
novelty     = −log2((purchases + 1) / (total_purchases + 1))
boost       = stock_boost × featured_boost × sale_boost × category_boost

final_score = log1p(popularity) × (novelty / 14) × boost

Комментарий к формулам

    Формулы выше не являются единственно правильными.

    Вы можете изменить их, если объясните:

        почему это уместно

        как это влияет на результат

        как вы избегаете утечки данных из будущего

Вопросы, на которые нужно ответить самостоятельно

Подготовьте краткие письменные ответы на следующие вопросы:

    Что такое оценки, извлекаемые из пользовательских событий, и как они могут неявно отражать намерение пользователя?

    Какие типы оценок можно использовать для расчета популярности товара?

    Как защититься от спамовых оценок и аномальной активности?

    Как учитывать ситуацию, когда товар давно присутствует в каталоге, но первое взаимодействие с ним произошло только спустя N дней?

    Какие возможные причины могут объяснить такой отложенный старт взаимодействий?

Основная задача

Нужно построить прототип сервиса или пайплайна, который подготавливает данные для индексации в Meilisearch.

Сервис должен:

    очистить и объединить данные из всех источников

    рассчитать признаки товаров и служебные поля для ранжирования

    сформировать документы для индекса

    загрузить эти документы в Meilisearch

    обеспечить получение результатов из поискового движка через запросы к индексу

Целевые страницы магазина

Нужно подготовить выдачу для следующих страниц:

    Новинки

    Распродажа

    Мужское

    Куртки

Для каждой страницы должно быть доступно не более 50 товаров.
Логика выдачи

Данные должны быть проиндексированы так, чтобы Meilisearch мог возвращать товары:

    в порядке по умолчанию по final_score

    с сортировкой по popularity

    с сортировкой по novelty

Что должно храниться в индексе

Каждый документ товара в индексе должен содержать достаточный набор полей для фильтрации, сортировки и выдачи.

Минимально стоит предусмотреть:

    product_id

    title

    category_id

    category_name

    brand

    price

    discount

    in_stock

    is_new

    is_sale

    gender

    popularity

    novelty

    final_score

    created_at

При необходимости можно добавить и другие поля, если они нужны для фильтрации или ранжирования.
Временное разделение и утечка из будущего

При расчете признаков и подготовке документов для индекса необходимо учитывать временной аспект.

Для даты расчета T можно использовать только:

    события, произошедшие не позже T

    состояние каталога и категорий, актуальное на момент T

    агрегаты, рассчитанные только по данным, доступным на момент T

Нельзя использовать:

    события из будущих периодов

    покупки, просмотры и другие действия, произошедшие после T

    агрегаты, рассчитанные по полному датасету без учета даты расчета

    признаки, восстановленные из более позднего состояния каталога

Использование таких данных считается утечкой из будущего и делает результат нереалистичным.
